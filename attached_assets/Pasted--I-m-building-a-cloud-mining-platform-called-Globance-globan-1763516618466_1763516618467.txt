5. Escrow & trade flows
Use/extend the current trades table to follow this state machine:
Statuses:
•	pending – trade created, waiting for buyer to pay.
•	payment_sent – buyer clicked “I have paid”, uploaded receipt.
•	released – seller confirmed, escrow released to buyer.
•	cancelled – cancelled before buyer payment.
•	expired – payment time window expired without buyer action.
•	disputed – buyer or seller clicked “Open dispute”.
•	dispute_resolved_buyer / dispute_resolved_seller – admin resolution.
Trade fields (extend existing):
•	id (uuid)
•	offer_id
•	buyer_id
•	seller_id
•	amount_usdt
•	price_fiat_per_usdt
•	total_fiat (DECIMAL)
•	fiat_currency_code
•	buyer_payment_method_id
•	seller_payment_method_id
•	status
•	escrow_amount_usdt
•	payment_deadline (timestamp)
•	dispute_reason (text, nullable)
•	dispute_notes (text/json for admin)
•	timestamps
Escrow behavior (VERY IMPORTANT):
We chose Option A: escrow deducted from main wallet.
When a BUYER takes a SELL offer:
1.	Validate seller’s wallet:
o	wallet.usdt_balance >= amount_usdt AND sufficient free balance considering other locked trades.
o	Implementation suggestion:
	Either use a dedicated escrow_balance column and move funds there,
	Or at least deduct from usdt_balance and track locked in trades.escrow_amount_usdt.
2.	On trade creation:
o	Subtract amount_usdt from seller’s usdt_balance.
o	Add amount_usdt to seller’s escrow_balance.
o	Set escrow_amount_usdt on trade.
o	Set status = 'pending'.
o	Set payment_deadline = now() + 30 minutes
(IMPORTANT: payment window is 30 minutes, not 15/20).
3.	Buyer view:
o	Sees seller’s bank/mobile-money details for the chosen payment method.
o	Can chat with seller via in-order chat (see #6).
o	After paying, buyer clicks “I have paid” and uploads payment screenshot / receipt.
4.	Seller view:
o	Sees buyer’s details and screenshot.
o	Can chat with buyer.
o	If payment is confirmed, clicks “Release USDT”:
	Move escrow_amount_usdt from seller’s escrow to buyer’s usdt_balance.
	Set trade status = 'released' or completed.
	Reduce remaining_amount_usdt from the offer.
5.	Expiry:
o	If buyer DOES NOT click “I have paid” within 30 minutes:
	Trade becomes expired.
	Escrow is released back to seller’s available balance.
o	Logic can be implemented by:
	A scheduled job (cron) or
	Lazy evaluation on next action (for now cron is fine, but you can also implement a backend endpoint triggered by existing cron system if we already have one).
6.	Cancellation rules:
o	Buyer can cancel only while status = 'pending' and before payment.
o	When cancelled:
	Escrow is unlocked → back to seller wallet.
	Trade status = 'cancelled'.
7.	Dispute:
o	After buyer clicks “I have paid” (payment_sent), either side can click “Open dispute”.
o	Trade status = 'disputed', dispute_reason saved from a simple text form.
o	Escrow stays locked.
o	Admin can see dispute data in admin panel and decide:
	Release funds to buyer → update wallets accordingly, set status = 'dispute_resolved_buyer'.
	Return funds to seller → escrow back to seller, set status = 'dispute_resolved_seller'.
