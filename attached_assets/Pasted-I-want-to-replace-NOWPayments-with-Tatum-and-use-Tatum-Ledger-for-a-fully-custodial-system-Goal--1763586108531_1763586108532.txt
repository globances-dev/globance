I want to replace NOWPayments with Tatum and use Tatum Ledger for a fully custodial system.

Goal:
Use Tatum Ledger + one or two hot wallets (TRC20 and/or BEP20) so that:
- Every user has an internal USDT balance (custodial)
- Deposits are automatic (on-chain -> Tatum -> webhook -> credit user)
- Withdrawals are automatic (user request -> Tatum send -> debit user)
- Minimum deposit = 5 USDT
- Minimum withdrawal = 5 USDT

Please implement the following:

────────────────────
1) Remove NOWPayments from deposit/withdraw flow
────────────────────
- Remove NOWPayments integration from:
  - Deposit address generation
  - Deposit webhook handler
  - Withdrawal implementation (if any)
- Keep all existing business logic:
  - mining packages
  - referral system
  - P2P marketplace
  - wallet balances (usdt_balance, escrow_balance, etc.)
We are only replacing the “how blockchain money enters/leaves the platform”.

────────────────────
2) Integrate Tatum Ledger (custodial model)
────────────────────
Use Tatum in “ledger / virtual account” mode, NOT one-chain-wallet-per-user.

Environment variables to set (use placeholders in code, I’ll fill values in Replit secrets):

- TATUM_API_KEY              → my Tatum API key
- TATUM_NETWORK_TRON         → e.g. "TRON"  (for TRC20 USDT)
- TATUM_NETWORK_BSC          → e.g. "BSC"   (for BEP20 USDT)
- TATUM_MASTER_WALLET_TRON   → hot wallet / ledger account ID for TRC20
- TATUM_MASTER_WALLET_BSC    → hot wallet / ledger account ID for BEP20
- APP_BASE_URL               → e.g. https://globance.app (for webhook URL)
- MIN_USDT_AMOUNT            → 5

Model:
- Do NOT create real blockchain wallets per user.
- Instead:
  - Create a Tatum ledger account (or internal mapping) per user for USDT.
  - Use a single (or small number of) blockchain hot wallet(s) per network for actual on-chain funds.
  - All user balances are tracked in Supabase + Tatum ledger style, not as separate private-key wallets.

────────────────────
3) User registration → attach Tatum ledger identity
────────────────────
On user registration (or first wallet access):

- Create and store per user:
  - tatum_ledger_id (or similar)
  - internal mapping so we can route deposits to the correct user
- If Tatum supports virtual accounts/ledgers directly, use those.
- This must be idempotent (no duplicate ledger accounts if user refreshes).

────────────────────
4) Deposit flow (automatic, min 5 USDT)
────────────────────
Behavior:
- Each user may have deposit addresses (TRC20, BEP20) generated via Tatum, or Tatum’s ledger routing.
- When USDT is sent on-chain, Tatum notifies our backend by webhook.
- Backend credits user if amount >= 5 USDT.

Implementation details:

4.1 Webhook:
- Create endpoint: POST /api/webhook/tatum-deposit
- Verify webhook authenticity using TATUM_API_KEY / signature as per Tatum docs.
- Extract:
  - user identifier (however we map from Tatum to our user_id)
  - amount in USDT
  - network (TRC20 or BEP20)
  - tx hash
- If amount < 5 USDT:
  - Insert a row in deposits table with status = 'failed' or 'below_min'
  - Do NOT credit user’s usdt_balance
  - Return success JSON so we don’t get retries forever
- If amount >= 5 USDT:
  - Insert row in deposits table with:
    - user_id, amount, network, txid, provider='tatum', status='completed'
  - Increase wallets.usdt_balance for that user by amount
  - Optionally insert a row into earnings_transactions (type='deposit') for audit.
- Return JSON { success: true }.

4.2 Frontend Wallet / Deposit UI:
- Keep the existing “Deposit USDT” screen.
- Replace any NOWPayments references with:
  - User sees their TRC20 and/or BEP20 deposit address (from Tatum).
  - Show “Minimum deposit: 5 USDT”.
- No change in UX flow: user copies address and sends from their exchange/wallet.

────────────────────
5) Withdrawal flow (automatic, min 5 USDT)
────────────────────
I want withdrawals to be fully automatic (no manual admin approval), but still safe.

Backend endpoint:
- POST /api/wallet/withdraw
  - Input: amount, network ("TRC20" or "BEP20"), destination address
  - Authenticated user only.

Logic:
1. Validate input:
   - amount >= 5 USDT (use MIN_USDT_AMOUNT env)
   - destination address is valid for selected network (basic regex or Tatum validation)
2. Check user balance:
   - Load user wallet from Supabase
   - If usdt_balance < amount → return error "Insufficient balance"
3. Calculate fee:
   - For now, assume platform_fee = 0 (no extra fee), we'll add later if needed.
4. Deduct from user:
   - Decrease wallets.usdt_balance by amount
   - Insert record into withdrawals table:
     - user_id, amount, network, to_address, status='pending' or 'processing'
5. Call Tatum:
   - Use TATUM_MASTER_WALLET_TRON or TATUM_MASTER_WALLET_BSC as the source
   - Send 'amount' USDT to destination address on selected network
   - Handle async response or success/failure.
6. On success:
   - Update withdrawals.status = 'completed'
   - Insert transaction into earnings_transactions (type='withdrawal')
7. On failure:
   - Option 1: revert user balance (add amount back)
   - Option 2: mark as 'failed' and log error clearly.

Frontend:
- Withdraw screen:
  - Show current USDT balance
  - Show “Minimum withdrawal: 5 USDT”
  - Network selector: TRC20 / BEP20
  - Address input
  - Amount input
  - Show confirmation dialog
- After successful request, show status and allow user to see history.

────────────────────
6) Keep everything else the same
────────────────────
- Mining system should use wallets.usdt_balance as source of funds for package purchases.
- Referral bonuses and daily mining income should still just credit wallets.usdt_balance.
- P2P escrow should still operate on internal balances (usdt_balance & escrow_balance).
- Only the “on-chain bridge” (deposit + withdrawal) is changing from NOWPayments to Tatum.
